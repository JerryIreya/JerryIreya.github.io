一个APP通常就是一个进程对应一个虚拟机

查看手机中所有进程

Android手机的底层是Linux系统，一个Android APP运行在一个Dalvik虚拟机里面 ，这个虚拟机对应Linux下的一个进程。

那就看一下手机上有哪些进程吧。

先进入shell：

使用ps命令列出所有的进程：

  • USER指当前进程所属的用户
  • PID指当前进程的id
  • PPID指当前进程的父进程的id
  • VSIZE指当前进程的虚拟大小
  • RSS指当前进程的常驻内存
  • NAME当前进程的名称，一般指包名

该命令会列出当前手机中所有的进程，如果你不好找你的APP的进程，可以使用grep过滤一下包名字符串：

可以看到我们的这个进程的PPID是1190(这是我的模拟器上的进程，跟你的可能不一样啊)，那么1190是哪个进程呢：

通过查看，知道是root进程，root进程也就是zygote进程，所有APP进程的父进程，所属用户是root。

APP的内存限制机制

每个APP分配的最大内存限制，随不同设备而不同

可以使用下面的代码获取一个设备分配个每个APP的最大内存限制：

ActivityManager activityManager = (ActivityManager) getSystemService(Context.ACTIVITY_SERVICE);
//一个设备分配给每个APP的内存空间，以M为单位
int memoryClass = activityManager.getMemoryClass();
//一个设备分配给每个APP的最大内存空间，以M为单位
//在manifest文件的<application>节点下声明  android:largeHeap="true" 属性时向系统申请large内存空间
//一般设备不好使，一般和getMemoryClass相等
int largeMemoryClass = activityManager.getLargeMemoryClass();

APP中吃内存大户： 图片

为什么要限制内存呢？

因为Android系统是多任务系统，一个系统上面有可能要运行很多APP，如果没有限制，一个APP占用了系统所有的内存，别的APP就无法运行了。

切换应用时后台APP清理机制

  • APP切换时使用的是LRU Cache机制，最近使用的放在最前面，最少可能被清理掉。排在末尾的最容易被回收掉。
  • 系统在清理APP之前会回调onTrimMemory(int level)方法,我们可以在该方法中把我们APP中比较占内存的自己清理掉，这样我们的APP所占用的系统的内存就会少，被系统回收的可能性就减小了。

onTrimMemory方法是Activity的方法。

监控内存的几种方法

上面介绍了使用shell去查看APP的内存情况，但是它只能查看一定时间上的内存使用情况，是静态的。

使用代码获取内存使用情况

使用Runtime获取内存使用情况：

//系统允许给当前APP分配的最大内存，和activityManager.getMemoryClass()一样
long maxMemory = Runtime.getRuntime().maxMemory();
//系统已经分配给当前APP的总内存
//要知道系统并不会一次性把每个APP的上限内存分配给你，因为如果你当前的APP只需要1M，而直接分配给你512M就会造成内存浪费
long totalMemory = Runtime.getRuntime().totalMemory();
//系统当前分配个你的APP的总内存中的剩余内存
//就是totalMemory - APP已经使用的内存
long freeMemory = Runtime.getRuntime().freeMemory();
//这些数值都是以字节为单位的

APP内存优化方法

数据结构优化

1、频繁字符串拼接会造成很多的中间字符串，造成大量内存使用，并且字符串频繁拼接效率是很低的。

用以下代码测试：

private final int ROW = 30;

private final int LENGTH = 300;
//30行300列的矩阵存放数据
private final int[][] container = new int[ROW][LENGTH];

private final Random random = new Random();

@Override
protected void onCreate (@Nullable Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.layout_second);
    ButterKnife.bind(this);

    initArr();
}

//初始化矩阵
private void initArr () {
    for (int i = 0; i < ROW; i++) {
        for (int j = 0; j < LENGTH; j++) {
            container[i][j] = random.nextInt();
        }
    }
}

@OnClick({ R.id.btnDoBLD, R.id.btnDoJoin })
public void onClick (View view) {
    switch (view.getId()) {
        case R.id.btnDoBLD:
            btnDoBLD();
            break;

        case R.id.btnDoJoin:
            btnDoJoin();
            break;
    }
}

//使用String拼接字符串
private void btnDoJoin () {
    Log.d(TAG, "btnDoJoin: ----------->   start");
    String str = "";
    for (int i = 0; i < ROW; i++) {
        for (int j = 0; j < LENGTH; j++) {
            str += container[i][j];
            str += ",";
        }
        Log.d(TAG, "btnDoJoin: join row : " + i);
    }
    Log.d(TAG, "btnDoJoin: ------------->   end");
    Log.d(TAG, "btnDoJoin: length: " + str.length());
}

//使用Stringbuilder
private void btnDoBLD () {
    Log.d(TAG, "btnDoBLD: ------------->  start");
    StringBuilder stringBuilder = new StringBuilder();
    for (int i = 0; i < ROW; i++) {
        for (int j = 0; j < LENGTH; j++) {
            stringBuilder.append(container[i][j]);
            stringBuilder.append(",");
        }
        Log.d(TAG, "btnDoBLD: bld row " + i);
    }
    Log.d(TAG, "btnDoBLD: ------------->  end");
    Log.d(TAG, "btnDoBLD: length: " + stringBuilder.length());
}

可以自己测试下，这里直接给出结论：

使用String拼接的效率远低于StringBuilder，差距非常大，内存开销也要高于StringBuilder，当然也要比StringBuilder更耗时。而且如果矩阵过大，还有可能造成ANR。

2、使用ArrayMap、SparseArray来替代HashMap

HashMap其实效率不是很高的，推荐使用ArrayMap、SparseArray。ArrayMap、SparseArray内存使用更少，因为他们对数据结构进行了优化；如果数据项比较多的话，这两者效率也比HashMap高。而且使用起来和HashMap调用方法一样。

3、内存抖动

内存抖动是因为变量使用不当引起的，比如说你突然间申请了很多变量(或者很多内存空间)，但是这些大量的变量或者空间很快就用完不用了，但是过了一会又去申请很多，然后很快又不用了…反反复复，这时，如果heap内存空间不够，GC就会发动对这些不用的内存进行回收。GC回收的时候所有线程都是要暂停的，所以如果这个过程在比较短的时间内重复出现，这个时候从内存Monitor上看就会看到锯齿一样的内存抖动。如果内存抖动时GC耗时比较长，就会造成APP不流畅。

以下代码演示：

private final int ROW = 10;
private final int ROW_LENGTH = 500000;

private final int[][] container = new int[ROW][ROW_LENGTH];
private final Random random = new Random();

@Override
protected void onCreate (@Nullable Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.layout_second);
    ButterKnife.bind(this);

    initContainer();
}

private void initContainer () {
    for (int i = 0; i < ROW; i++) {
        for (int j = 0; j < ROW_LENGTH; j++) {
            container[i][j] = random.nextInt();
        }
    }
}


@OnClick({ R.id.btnDoBLD, R.id.btnShake })
public void onClick (View view) {
    switch (view.getId()) {
        case R.id.btnDoBLD:
            break;

        case R.id.btnShake:
            doShake();
            break;
    }
}

/**
 * 演示内存抖动
 */
private void doShake () {
    //str如果放到这里就不会造成内存抖动，这时会重复利用str
    // String[] str = new String[ROW_LENGTH];
    for (int i = 0; i < ROW; i++) {
        //内存抖动主要原因就是在这里
        //这里str的作用域只在外层循环之内，执行下次循环时，上层循环的str就作废了
        //优化很简单，将str的声明放到for循环之外，重复利用str就可以了
        String[] str = new String[ROW_LENGTH];
        for (int j = 0; j < ROW_LENGTH; j++) {
            str[j] = String.valueOf(container[i][j]);
        }
    }
}

小知识

  • 再小的Class都要耗费0.5KB
  • HashMap中的一个Entry需要额外占用32B(节点指针)

对象复用

对象复用的目的就是使内存消耗少一点。

  • 最好复用系统自带的资源，系统已经有了，内存占用就不会放到你的账上。
  • ListView/GridView里的ContentView复用。
  • 尽量避免在onDraw()方法里面执行对象的创建。空间只要一改变就会调用onDraw，onDraw是会频繁执行的。

避免内存泄漏

内存泄漏：由于代码瑕疵，导致某块内存，虽然停止不用了，但依然被其它东西引用着，导致GC没法对它回收

内存泄漏会导致剩余可用Heap越来越少，频繁触发GC。

Activity是很容易造成内存泄漏的。

引用上下文时，如果可以用Application Context就用Application Context去替代Activity Context。因为Application Context存在于整个App的生命周期里。

注意Cursor对象及时关闭。

OOM问题分析

OOM出现的绝大多数场景都是和图片相关的。

强引用、弱引用的意义

强引用

我们平时常用的都是强引用，强引用在其声明周期内是不会被回收的。

弱引用 SoftReference

弱引用在其声明周期内，如果Heap内存不足，是可以被回收的。

弱引用使用：

//一个Bitmap的弱引用
private SoftReference<Bitmap> bitmapSoftReference;

@Override
protected void onCreate (@Nullable Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.layout_second);
    ButterKnife.bind(this);

    bitmapSoftReference = new SoftReference<>(BitmapFactory.decodeResource(getResources(), R.mipmap
            .ic_launcher));

    //从弱引用中取出Bitmap对象
    Bitmap bitmap = bitmapSoftReference.get();

优化OOM问题的方法

1、注意临时Bitmap对象的及时回收。

2、避免Bitmap的浪费。

3、try/catch某些大内存分配的操作。

4、加载Bitmap：缩放比例、解码格式、局部加载。

第四条可以解决绝大部分因图片引起的OOM问题。
