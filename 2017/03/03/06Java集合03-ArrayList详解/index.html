ArrayList介绍

ArrayList是一个数组队列，相当于动态数组。与Java中的数组相比，它的容量能动态增长。 它继承与AbstractList，实现了List，RandomAccess，Cloneable，java.io.Serializable这些接口。

ArrayList继承了AbstractList，实现了List。它是一个数组队列，提供了相关的增加、删除、修改、遍历等功能。

ArrayList实现了RandomAccess接口，即提供了随机访问功能。RandomAccess是java中用来被List实现，为List提供快速访问功能的。在ArrayList中，我们即可以通过元素的序号快速获取元素对象，这就是快速随机访问。

ArrayList实现了Cloneable接口，即覆盖了函数clone()，能被克隆。

ArrayList实现了java.io.Serializable接口，这意味着ArrayList支持序列化，能通过序列化去传输。

ArrayList包含了两个重要的对象：elementData和size

1、elementData是“Object[]类型的数组”，它保存了添加到ArrayList中的元素。实际上，elementData是个动态数组，我们能通过构造函数ArrayList(int initialCapacity)来指定它的初始容量为initialCapacity；如果通过不含参数的构造函数ArrayList()来创建ArrayList，则elementData的容量默认是10.elementData数组的大小会根据ArrayList容量的增长而动态的增长，具体的增长方式，参考源码中的ensureCapacity()函数。

2、size则是动态数组的实际大小。

ArrayList遍历方式

有三种： 随机访问(普通for循环)、迭代器、增强for

其中随机访问(普通for循环，通过索引访问)效率最高，使用迭代器效率最低，因为要内部要维护一个Iterator对象；foreach的本质也是迭代器；在使用迭代器遍历ArrayList的时候，如果使用ArrayList的方法修改了集合的长度(调用了ArrayList的add或remove等其他能修改集合长度的方法)，那么迭代器就会抛出ConcurrentModificationException，这就是fail-fast机制。

fail-fast机制

如果使用普通for遍历一个ArrayList，在遍历的同时可以通过调用ArrayList的add、remove等方法去改变集合的长度，虽然不会报错，但是会有意想不到的结果：

/** 
 * 这种遍历有可能会遗漏某个元素,因为删除元素后List的size在 
 * 变化，元素的索引也在变化，比如你循环到第2个元素的时候你把它删了， 
 * 接下来你去访问第3个元素，实际上访问到的是原先的第4个元素。当访问的元素 
 * 索引超过了当前的List的size后还会出现数组越界的异常，当然这里不会出现这种异常， 
 * 因为这里每遍历一次都重新拿了一次当前List的size。 
 */  
public void listRemove2() {  
    List<Student> students = this.getStudents();  
    for (int i=0; i<students.size(); i++) {  
        if (students.get(i).getId()%3 == 0) {  
            Student student = students.get(i);  
            students.remove(student);  
        }  
    }  
}

Java为了避免这种意想不到的情况出现，就出现了fail-fast(快速失败)机制。在集合里面有一个成员变量modCount来记录集合长度改变的次数，就是在集合的add、remove、trimToSize等改变集合长度的方法里面会对该值进行+1操作；当我们使用Iterator去遍历一个集合的时候，在生成Iterator对象时会记录一个expectedModCount=modCount；每次调用Iterator对象的next()方法时都会检查这两个值是否相等；因为modCount只有在集合的长度发生变化的时候才有可能改变，一旦它改变了，那么expectedModCount就和modCount不相等了，就直接抛出ConcurrentModificationException。用抛异常的方式去避免意想不到的结果

看看官方的解释：

ArrayList的iterator和listIterator方法返回的迭代器是快速失败的：在创建迭代器之后，除非通过迭代器自身的remove或add方法从结构上对列表进行修改，否则在任何时间以任何方式对列表进行修改，迭代器都会抛出ConcurrentModificationException。因此，面对并发修改，迭代器很快就会完全失败，而不是冒着在将来某个不确定的时间发生任意不确定行为的风险。

为了实现fail-fast机制，ArrayList从AbstractList继承了protected transient int modCount字段，表示已经从结构上修改此列表的次数。从结构上修改是指更改列表的大小，或者以其他方式打乱列表，使正在进行的迭代产生错误的结果。

此字段由iterator和listIterator方法返回的迭代器和列表迭代器实现来使用。如果意外更改了此字段中的值，则迭代器或列表迭代器将抛出ConcurrentModificationException来响应next、remove、previous、set或add操作。在迭代期间面临并发修改时，它提供了快速失败行为，而不是非确定性行为。

子类是否使用此字段是可选的。如果子类希望提供快速失败迭代器或列表迭代器，则它只需在其add()和remove()方法(以及它所重写的、导致列表结构上修改的任何其他方法)中增加此字段。对add()或remove()单个调用此字段添加的数量不得超过1，否则迭代器将抛出虚假的ConcurrentModificationException。如果某个实现不希望提供快速失败迭代器，则可忽略此字段。

ArrayList遍历的同时修改集合中的元素(改变ArrayList的长度)

上面讲了，普通for循环遍历集合的时候最好不要修改集合的长度，因为异常不可怕，可怕的是意想不到的结果。

  1. 虽然普通for循环效率很高，但是如果在遍历的过程中有修改集合长度的操作，容易引起意想不到的结果，所以普通for循环不适用；当然了，如果只是遍历集合，在遍历的过程中并没有改变集合长度的操作，用普通for循环还是很合理的。
  2. 使用Iterator迭代器遍历集合(包括iterator方法生成的迭代器对象，或者foreach)时，如果调用了集合的add、remove、trimToSize等改变集合长度的方法又会抛出并发修改异常。

如果既不想引起意想不到的结果，又不想抛出异常(前提是在遍历的同时有改变集合长度的操作，否则直接用普通for循环即可)，我们可以有以下几种解决方案：

1、使用Iterator的remove和add方法：

// 删除/修改元素 
Iterator<String> it = list.iterator(); 
while(it.hasNext()){ 
    String ele = it.next(); 
    it.remove(); //调用Iterator对象的remove方法。
}

2、使用CopyOnWriteArrayList替代ArrayList，CopyOnWriteArrayList可以在使用迭代器遍历的同时调用集合的add或remove方法改变集合长度，而不引起并发修改异常。

//list是CopyOnWriteArrayList类型的。
for(String item : list){ 
    list.remove(item); 
}

使用CopyOnWriteArrayList增加或删除元素的时候，会先将内部维护的数组复制一份，对副本进行操作，然后把之前的数组变量指向副本数组。但是在Iterator迭代器的生存期内所遍历的数组一直都是之前的数组(这就是快照(snapshot)),所以不会出现异常。CopyOnWriteArrayList的add和remove方法都是加锁的，即线程安全和同步的，不用担心并发访问带来的问题。

注意：CopyOnWriteArrayList内部的Iterator对象不支持remove和add：

public void add(E object) {
        throw new UnsupportedOperationException();
    }

public void remove() {
        throw new UnsupportedOperationException();
    }

所以使用迭代器遍历CopyOnWriteArrayList的时候需要调用CopyOnWriteArrayList集合对象的add和remove方法对集合进行修改。

ArrayList并发修改

ArrayList是线程不同步的，也就是线程不安全的。

多线程同时访问一个数据免不了会产生问题，更何况是一个集合，有那么多改变集合的方法。

ArrayList非线程同步，所以效率就比较高；如果是单线程就使用ArrayList。

Vector和ArrayList几乎一样，只不过是在所有可能改变集合数据的方法上面都加上了锁，这也就意味着Vector的效率不如ArrayList，因为它需要额外的开销去维护锁对象。

CopyOnWriteArrayList也是线程同步的。多线程里也可以使用。同样是效率低。另外该类在所有改变集合struct(结构)的方法里都会复制出原始数组的副本，然后去修改副本,修改完之后再将之前的数组变量指向副本。所以该类如果执行大量的增删操作，会占用很大的内存空间(改变一次集合长度，就要复制出来一个副本)。CopyOnWriteArrayList适合用在遍历操作的数量大大超过可变操作的数量的情况下。
